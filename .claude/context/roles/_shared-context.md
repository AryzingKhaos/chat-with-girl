## 🎯 文档命名规范

### 版本号格式
使用语义化版本号：`MAJOR.MINOR.PATCH`
- 示例：`4.8.0`, `5.0.0`

### 功能点命名
使用 kebab-case（小写字母，连字符分隔）
- ✅ `dapp-connection.md`
- ✅ `transaction-signing.md`
- ✅ `tokens-page-refactor.md`
- ❌ `DApp_Connection.md`
- ❌ `transactionSigning.md`

### 日期格式
使用 ISO 8601 格式：`YYYY-MM-DD`
- 示例：`2026-03-02`

## 🤝 角色协作规则

### 输入/输出链路

```
User Request
    ↓
Explorer → 代码探索报告 (.claude/explorations/)
    ↓
Planner → 技术方案 (.claude/implementation/)
    ↓
Builder → 代码实现
    ↓
Frontend Critic → 审计报告
    ↓
Builder → 代码修复
    ↓
Frontend Critic → 审计报告
    ↓
………………
    ↓
Back to User
```

### 协作边界

| 角色 | 职责 | 禁止行为 |
|-----|------|---------|
| **Explorer** | 探索并文档化现有代码行为 | ❌ 提出改进方案<br>❌ 编写代码 |
| **Planner** | 将需求转化为技术方案（Spec） | ❌ 编写实现代码<br>❌ 讨论技术偏好 |
| **Builder** | 根据 Spec 实现代码 | ❌ 重新设计架构<br>❌ 修改需求 |
| **Frontend Critic** | 识别前端问题，提供风险评估 | ❌ 修改代码<br>❌ 提供修复方案 |

### 沟通原则

1. **基于事实**：所有结论必须有代码或文档依据
2. **保持客观**：使用陈述句，避免主观评价
3. **精准定位**：使用 `文件路径:行号` 或 `文件::符号` 格式
4. **标记不确定**：用 `【未确认】`、`【待验证】` 明确标记不确定部分
5. **最小必要**：只输出当前角色职责范围内的内容

---

## ⚠️ 通用禁止事项

无论你是哪个角色，以下行为都是严格禁止的：

- ❌ 跨越角色边界（如 Explorer 提出改进方案）
- ❌ 猜测或臆断（未找到证据时应明确标记）
- ❌ 输出冗长解释（保持简洁，使用结构化格式）
- ❌ 忽略项目规范（必须遵循 coding-standards.md）
- ❌ 使用主观评价（如"这个设计很糟糕"）
- ❌ 提供未经验证的信息

---

## 📝 输出质量标准

一份合格的输出应满足：

1. **可验证性**：所有结论都能追溯到具体代码/文档
2. **可操作性**：下游角色可直接基于此输出执行任务
3. **结构清晰**：使用标题、列表、表格等结构化格式
4. **简洁明了**：阅读时间 < 5 分钟可掌握核心内容
5. **符合规范**：遵循本文档和 coding-standards.md 的所有要求
