## TodoList 编写规范

每个功能点文档必须包含详细的 TodoList，帮助 Builder 按步骤实施。

### TodoList 结构

```markdown
## Todo List

### 分类 1（如：类型定义）
- [ ] 具体任务 1
- [ ] 具体任务 2
- [x] 已完成任务（使用 [x] 标记）

### 分类 2（如：组件开发）
- [ ] 具体任务 3
- [ ] 具体任务 4

### 分类 3（如：功能测试）
- [ ] 测试场景 1
- [ ] 测试场景 2
```

### TodoList 要求

1. **任务分类**：按功能模块、实施阶段分组（如：类型定义、工具函数、组件开发、测试等）
2. **任务粒度**：每个任务应清晰、可执行、可验证
3. **任务描述**：使用动词开头，明确说明要做什么（如："创建 XXX 文件"、"实现 XXX 功能"、"测试 XXX 场景"）
4. **优先级标注**：对于有严格顺序要求的任务，使用编号（Step 1、Step 2）或标注（⚠️ 最优先）
5. **状态标记**：
   - `[ ]` - 待完成
   - `[x]` - 已完成
   - 可选：使用 ✅ ❌ 📝 ⚠️ 等 emoji 辅助标记
6. **更新维护**：
   - 新增功能点时，必须添加完整的 TodoList
   - 修改功能点时，必须同步更新 TodoList（新增、删除、修改相关任务项）
   - 功能点有依赖关系时，在 TodoList 中明确标注

### TodoList 示例

```markdown
## Todo List

### Step 1: 测试脚本开发（最优先）⚠️
- [ ] 创建测试目录结构
- [ ] 准备测试数据（3 个样本）
- [ ] 编写测试脚本
- [ ] 运行测试并生成报告
- [ ] 验证测试结果

### Step 2: 类型定义
- [ ] 创建 `src/types/xxx.ts`
- [ ] 定义 `InterfaceA` 接口
- [ ] 定义 `InterfaceB` 接口
- [ ] 定义 `TypeC` 类型

### Step 3: 组件开发
- [ ] 创建 `ComponentA.tsx`
- [ ] 实现组件 A 的核心功能
- [ ] 创建 `ComponentB.tsx`
- [ ] 实现组件 B 的样式

### Step 4: 功能测试
- [ ] 测试：正常场景 A
- [ ] 测试：边界情况 B
- [ ] 测试：错误处理 C

### 代码质量
- [ ] 所有注释使用中文
- [ ] 无 TypeScript `any` 类型
- [ ] 通过 ESLint 检查
```

## 修改功能点时的 TodoList 更新规则

当需要修改已有功能点文档时：

1. **评估影响范围**：确定修改影响哪些实施步骤
2. **更新现有任务**：修改受影响的任务项描述
3. **新增任务项**：如果修改引入新的工作，添加新的任务项
4. **标记新增部分**：使用日期或版本号标注新增的任务分类（如：`### Prompt 优化 📝 (2026-03-03)`）
5. **保留已完成任务**：不要删除已标记为 `[x]` 的任务，保持历史记录
6. **调整优先级**：如果修改改变了实施顺序，重新标注优先级

### 修改示例

**场景**：优化 OCR Prompt

**原 TodoList**：
```markdown
### 结构化 OCR 测试验证 ✅
- [x] 创建测试脚本
- [x] 测试对话人物识别
- [x] 测试昵称提取
```

**更新后的 TodoList**：
```markdown
### 结构化 OCR 测试验证 ✅
- [x] 创建测试脚本
- [x] 测试对话人物识别
- [x] 测试昵称提取

### Prompt 优化 📝 (2026-03-03)  ← 新增部分
- [ ] 更新 Route.ts 中的 Prompt
- [ ] 添加气泡特征判断标准
- [ ] 重新运行测试验证效果
- [ ] 对比优化前后的准确率
```

不要解释设计过程。
不要讨论替代方案。
只输出最终 Spec。
